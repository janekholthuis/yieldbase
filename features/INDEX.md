> **PROJ-7 Selbstauskunft-OHNE-Login (Token-Link) — Security-Audit (QA 2026-06-15):** Red-Team-Pass der **unauthentifizierten** Fläche `/selbstauskunft/[token]` + `src/lib/actions/selbstauskunft-public.ts` (alle Reads/Writes via Service-Role-Client, strikt auf den Token-Kunden gescopt). **Solide Basis:** Token = `kunden.selbstauskunft_token` DEFAULT `gen_random_uuid()` (UUIDv4, 122 Bit, NOT NULL, ≠ `id`) → nicht brute-forcebar; **kein IDOR** (kein client-gelieferter `kunde_id` wird vertraut, Immobilien-Subform an `selbstauskunft_id` des Token-Kunden gebunden); IP wird vor `inet`-Cast via `sanitizeIp()` validiert; Invalid-Token → generische Meldung (kein Oracle); Token nur dem verwaltenden VP sichtbar (`KundeDetailView`); die `submit_selbstauskunft`-RPC (Advisor-WARN „anon-executable") ist die **authentifizierte** Variante und fail-safe (`where user_id = auth.uid()` → für anon leer → RAISE). **Keine Critical/High.** **Findings:** (1) **BUG-S1 (Medium)** — keine serverseitige Immutability/Expiry: `submitSelbstauskunftByToken` akzeptiert & überschreibt eine bereits `eingereicht`-e (unterschriebene) Selbstauskunft erneut (`upsert onConflict kunde_id`, nur Client-Flag `alreadySubmitted`); Token läuft nie ab / ist nicht Single-Use. Ein zurückbehaltener/geleakter Link erlaubt Manipulation eines rechtlich unterschriebenen Finanz-Dokuments + stilles Neuberechnen der Bonität (`max_finanzierbar` etc.). Empfehlung: Submit bei `status='eingereicht'` ablehnen oder versionieren; Token nach Submit rotieren/expiren. (2) **BUG-S2 (Low)** — Public-Token-Seite ohne `robots: noindex`/`X-Robots-Tag`; Magic-Link-URLs sollten nicht indexierbar sein. (3) **BUG-S3 (Low/Obs.)** — kein Rate-Limit auf den Public-Actions (Token-Brute-Force dank UUIDv4 unrealistisch; ein Token-Inhaber kann aber Writes spammen). Tests: 29 grün (`selbstauskunft` + `bonitaet`). E2E n/a (lokal kein Token-Seed). Status bleibt **Deployed** (Medium blockiert nicht, aber Fix empfohlen).
